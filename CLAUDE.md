# Furton Coverage

Automated earnings-coverage desk on ~10 names, built with Claude Code on **free, public SEC
EDGAR data** (no paid vendor — reworked off Daloopa 2026-07-08). Read PLAN.md before doing
anything — it defines the phases, session pairings, and path ownership.

## Rules for every session

- **You are one of a pair.** Another Claude Code session may be working in this repo right now.
  Write ONLY inside the paths your kickoff prompt says you own. Shared proposals go in
  `notes/_inbox/`, never directly into another session's files.
- **Commits:** prefix with your session tag (`[S3] ...`), stage only your owned files (never
  `git add -A`). On push/commit races: `git pull --rebase`, retry.
- **End-of-session:** commit, push, append a 3-line handoff to `notes/_inbox/handoffs.md`
  (what you did / what's unfinished / what the next session must know).
- **Fail loud.** Missing data, an EDGAR fetch/parse error, or an unextractable guidance figure is
  a stop-and-report, never a silently degraded or invented output. (Same philosophy as Furton
  Research's vote-integrity work.)

## Data spine (this is what replaced Daloopa)

- **All SEC data flows through `scripts/edgar.py`** — the only module that touches the network.
  It owns: ticker→CIK, XBRL companyfacts fetch/parse (fundamentals + KPIs), submissions/filing
  retrieval, and finding a company's earnings 8-K (Exhibit 99.1). Everything else reads its cache.
- **SEC etiquette is non-negotiable:** User-Agent `Furton Coverage nicholas@furton.com` on every
  request, ≤10 req/s, backoff, cache raw JSON in `data/raw/` (gitignored) — never refetch.
- Fundamentals ← XBRL companyfacts. Reported quarterly numbers + management guidance ← the
  earnings 8-K press release. Consensus is NOT on EDGAR → web-sourced, cited, color only.
- Everything we publish is EDGAR-sourced (public domain) → fully ours to publish/sell, no ToS.

## Conventions

- Notes: `notes/<TICKER>/<YYYY-MM-DD>_<type>.md`, type ∈ preview | flash | review | initiation.
  Required frontmatter: `ticker, type, event_date, published_at`.
- Models: `models/<TICKER>/<TICKER>_model_<YYYY-MM-DD>.xlsx`, built by `scripts/build_model.py`
  from companyfacts XBRL via the xlsx skill.
- Scorecard is sharded PER TICKER: `scorecard/calls/<TICKER>.jsonl`,
  `scorecard/guidance/<TICKER>.jsonl` — parallel sessions on different tickers never touch the
  same file. `scorecard/summary.json` is generated by score.py only.
- Scorecard entries carry `source_note` (the note that made the call); guidance entries also
  carry `source_filing` (the EDGAR 8-K URL the guidance came from) so every figure is auditable.
- **Grading benchmark = management guidance** (itself an EDGAR artifact; consensus is cited color
  only). Numeric calls are graded deterministically by score.py — no LLM in the grading path;
  Claude never grades its own quantitative calls.
- **GAAP/non-GAAP basis:** guidance & grading actuals come from the 8-K press release (company's
  own basis); XBRL companyfacts (GAAP) is for the model/history only. Every call & guidance
  record carries a `basis` field; score.py refuses to grade across bases. Getting this wrong
  silently biases the scorecard.
- **Tests:** edgar.py has golden-fixture parser tests, score.py has grading unit tests (both in
  tests/). `pytest` must pass before a merge gate closes.
- **Flash (T+0) source = the earnings 8-K (Exhibit 99.1) from EDGAR + web for color.** A missed
  T+0 becomes an honestly-timestamped delayed flash — never backdated; a skipped print gets a
  dated skip note.
- `docs/` is generated by `scripts/build_site.py` — never hand-edit it.
- Site preview: port **8802** (8801, 8899, 8765, 8803 are taken by other Furton projects).
- Every published page carries the educational / not-investment-advice disclaimer.

## Quality toolkit (use these, don't just know about them)

- Wrote script/pipeline code this session? Run **/code-review** on your diff before the final
  commit, and use the **verify** skill to exercise what you built end-to-end — run it for real
  against live EDGAR, don't just eyeball it.
- Publishing a note? It must pass the **adversarial fact-check workflow** (parallel agents
  re-verify every numeric claim against its cited EDGAR document) built into the /preview /flash
  /review /digest skills. No note goes public unchecked.
- Building or changing a project slash command? Use the **skill-creator** skill.

## Environment notes (Windows)

- Python: run via `py`; set `PYTHONUTF8=1` (console is cp1252 — ✓/→ symbols crash otherwise).
- Word/PDF: no LibreOffice; convert docx→PDF via Word COM in a PowerShell job.
- EDGAR is a public no-auth HTTP API → works fully headless in scheduled/cloud runs (unlike a
  vendor connector). Lean into that for automation.
- `gh` is installed and authed as Nick-Furton.

## Status board — UPDATE AT END OF SESSION

There is a shared cross-project status board at `..\status-dashboard.html` (open the file
directly — no server). Whenever you complete meaningful work on Furton Coverage (close a gate,
finish a session/phase, change metrics/next-actions/blockers), update the `DATA` object for the
`coverage` project inside that file before ending: bump `lastUpdated`, adjust the phase `status`,
`percent`, `currentPhase`, `metrics`, `nextActions`, `blockers`, and add a `milestones` line. The
file has a `HOW TO UPDATE` banner in its `<script>`.
