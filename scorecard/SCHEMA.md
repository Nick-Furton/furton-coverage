# Scorecard schemas (Session 4)

The scorecard is **sharded per ticker** — one JSONL file per ticker under each of
`scorecard/calls/` and `scorecard/guidance/`. Sharding is what lets parallel sessions
on different tickers coexist: a `/flash NVDA` run only ever appends to
`scorecard/calls/NVDA.jsonl`, so it never races a `/flash AMD` run on a different file
(PLAN §3, §4). `scorecard/summary.json` is generated by `scripts/score.py` **only** —
never hand-edit it.

- One JSON object per line. Append-only. A published call is never rewritten; a
  correction is a new dated record (PLAN §7 — "never quietly edit a published call").
- Every record is timestamped and carries a `basis` (`gaap` | `non_gaap`).
- The **filename ticker must equal the record's `ticker`** — `score.py` fails loud if a
  record is filed in the wrong shard (a mis-sharded call would silently skew a ticker).
- Dummy seed rows carry `"dummy": true`. `score.py` scores them by default so the site
  pair can build against real structure; `--exclude-dummy` drops them, and Merge Gate 3
  deletes them once real batch-1 content lands.

Both stores fail loud (`MalformedRecordError`) on missing/ill-typed required fields, an
invalid `basis`, an unknown `kind`, or a mis-filed shard — same philosophy as Furton
Research's vote parser. There is **no LLM in the grading path**: `score.py` compares call
vs. actual in code (PLAN §3).

---

## The GAAP / non-GAAP basis rule (the §3 trap)

Every call and every guidance record stores the `basis` it is stated on, and every
`actual` block **also** stores its own `basis`. Before any numeric comparison,
`score.py` checks `record.basis == actual.basis`; if they differ it raises
`BasisMismatchError` — **it does not silently score**. This is the deliberate guard
against grading a non-GAAP guidance call (adjusted EPS/revenue — what companies guide to
and the market reacts to) against a GAAP companyfacts actual. Companyfacts/XBRL is GAAP
and is for the model and history; the 8-K press release is the grading source and carries
the company's own (usually non-GAAP) basis.

> `unit` is single-source (top-level only) — the `actual` block does not repeat it, so
> there is no unit to drift. `basis` is deliberately duplicated onto `actual` precisely
> so the trap is detectable at grade time.

## `scorecard/calls/<TICKER>.jsonl` — our calls

One record = one **falsifiable, machine-gradable call** we made in a note. Session 3's
note frontmatter must carry these fields so `score.py` can grade in code without asking
Claude (wired at Merge Gate 2).

| field | req | type | notes |
|---|---|---|---|
| `id` | ✓ | str | stable unique id, e.g. `NVDA-2026Q3-preview-rev` |
| `ticker` | ✓ | str | uppercase; **must equal the shard filename** |
| `timestamp` | ✓ | str | ISO-8601, when the call was made (`published_at` of the note) |
| `call_type` | ✓ | str | `preview` \| `flash` \| `review` \| `initiation` — drives hit-rate-by-type |
| `period` | ✓ | str | the fiscal period the call is **about**, e.g. `FY2026Q3` — the over-time axis |
| `metric` | ✓ | str | e.g. `revenue`, `eps_adj`, `gross_margin`, `data_center_revenue` |
| `unit` | ✓ | str | e.g. `USD_M`, `USD`, `eps_usd`, `pct`, `ratio` — apples-to-apples |
| `basis` | ✓ | str | `gaap` \| `non_gaap` |
| `source_note` | ✓ | str | repo-relative path to the note that made the call (audit link) |
| `call` | ✓ | obj | the prediction — one of the four `kind`s below |
| `actual` | – | obj | realized outcome, filled when the print lands (see below); absent ⇒ **pending** |
| `higher_is_better` | – | bool | default `true`; set `false` for cost/capex/opex metrics |
| `confidence` | – | num | `0.0–1.0`; used for the calibration curve (optional) |
| `dummy` | – | bool | seed rows only |
| `rationale` | – | str | free text, ignored by the grader |

### `call` kinds

- **`direction`** — we predict where the actual lands vs. a benchmark band (guidance,
  consensus-color, or prior period):
  ```json
  {"kind": "direction", "value": "beat", "benchmark": {"kind": "range", "low": 194000, "high": 199000}}
  ```
  `value` ∈ `beat` | `met` | `miss`. `benchmark` is a `range` (`low`/`high`) or a
  `point` (`value`, optional `tolerance` default 0). **Hit** iff our `value` equals the
  actual's classification vs. that band (with `higher_is_better` applied).
- **`range`** — we predict the actual lands in `[low, high]`:
  ```json
  {"kind": "range", "low": 45000, "high": 47000}
  ```
  **Hit** iff `low ≤ actual ≤ high`.
- **`point`** — we predict a value with a tolerance:
  ```json
  {"kind": "point", "value": 5.12, "tolerance": 0.05}
  ```
  **Hit** iff `|actual − value| ≤ tolerance` (tolerance default 0).
- **`qualitative`** — an explicitly non-numeric call:
  ```json
  {"kind": "qualitative", "value": "AWS reacceleration continues"}
  ```
  `score.py` **counts** these but does **not** grade them (no LLM in `score.py`). A
  separate judgment pass (Session 3's fact-check workflow) handles qualitative calls.

### `actual` block (shared by both stores)

```json
{"value": 196300, "basis": "non_gaap", "asof": "2026-10-30", "source_filing": "https://www.sec.gov/Archives/edgar/data/1018724/.../ex991.htm"}
```

| field | req | type |
|---|---|---|
| `value` | ✓ | num |
| `basis` | ✓ | `gaap` \| `non_gaap` — **must match** the record's `basis` or grading raises |
| `asof` | – | str (ISO date) |
| `source_filing` | – | str (EDGAR URL of the 8-K the actual was read from) |

## `scorecard/guidance/<TICKER>.jsonl` — management guidance vs. actuals

One record = one piece of **management's own quantitative guidance** and how it later
landed. This is the grading benchmark itself (PLAN §3): guidance is an EDGAR artifact
(the prior quarter's 8-K EX-99.1), so the whole benchmark is free, verifiable, and
republishable. Backfilled by Session 7 (last 4 quarters × 10 names).

| field | req | type | notes |
|---|---|---|---|
| `id` | ✓ | str | e.g. `AMZN-2026Q3-guid-netsales` |
| `ticker` | ✓ | str | **must equal the shard filename** |
| `timestamp` | ✓ | str | ISO-8601 — when this record was written |
| `metric` | ✓ | str | e.g. `net_sales`, `operating_income`, `eps_adj` |
| `unit` | ✓ | str | as in calls |
| `basis` | ✓ | str | `gaap` \| `non_gaap` — the basis **management guided on** |
| `period` | ✓ | str | the fiscal period the guidance is **for**, e.g. `FY2026Q3` |
| `guided_at_period` | ✓ | str | the quarter whose 8-K **issued** this guidance, e.g. `FY2026Q2` |
| `guidance` | ✓ | obj | `{"kind":"range","low":..,"high":..}` or `{"kind":"point","value":..,"tolerance":..}` |
| `source_filing` | ✓ | str | EDGAR 8-K URL the **guidance** was read from (audit link) |
| `actual` | – | obj | reported result (period N+1's 8-K); absent ⇒ **pending** |
| `higher_is_better` | – | bool | default `true`; `false` for cost/capex guides |
| `dummy` | – | bool | seed rows only |

**Grading** (deterministic): classify `actual.value` against the guidance band —
`above` → **beat**, `within` → **met**, `below` → **missed** (labels flip when
`higher_is_better` is `false`, e.g. capex guided low and coming in high is a *miss*).
Basis guard applies exactly as above.

A record that genuinely reflects "no quantitative guidance given" is **not** invented as a
number — it is simply not written as a guidance record (PLAN §7). Absence is honest;
a fabricated figure is not.

## `scorecard/summary.json` (generated)

Produced by `scripts/score.py` — shape documented in that file's module docstring. Top
level: `generated_at`, `totals`, `our_calls` (overall / by_call_type / by_metric /
calibration / over_time), `guidance_accuracy` (overall / by_ticker / over_time),
`by_ticker`, `dummy` counts, and `warnings`. The site generator (Session 5) reads only
this file.
